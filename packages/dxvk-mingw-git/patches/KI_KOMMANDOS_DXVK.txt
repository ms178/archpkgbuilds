# Project Context — Read This Before Analyzing

**DXVK** translates D3D9/D3D10/D3D11 API calls into Vulkan. It is written in **C++20** (with `-std=c++20`, heavy use of `std::atomic`, `std::mutex`, `std::vector`, `small_vector`, COM-style ref counting via `Com<T>` / `Rc<T>`, and a custom command-stream architecture with a dedicated **CS thread**). Critical hot paths include:
- **Command stream (CS) recording and replay**: `DxvkCsChunk`, `DxvkCsThread` — serialization of draw/dispatch commands.
- **State tracking and dirty flags**: `DxvkContext` and its `DxvkContextState` — tracking which Vulkan state needs re-emission per draw.
- **Descriptor management**: `DxvkDescriptorPool`, `DxvkDescriptorManager` — descriptor set allocation and update batching.
- **Shader/pipeline compilation**: `DxbcCompiler` (DXBC→SPIR-V), `DxvkGraphicsPipeline`, `DxvkShaderPipelineLibrary` — pipeline creation and caching.
- **Resource binding**: `D3D11DeviceContext::bindResources` patterns — constant buffer, SRV, UAV, sampler binding and null-check overhead.
- **Barrier and layout tracking**: `DxvkBarrierSet`, `DxvkBarrierTracker` — accumulating and flushing barriers.
- **Draw/dispatch call overhead**: Per-draw CPU cost including state validation, dirty-flag checks, and VkCmdDraw emission.

**vkd3d-proton** translates D3D12 into Vulkan. It is written in **C17** (with POSIX threading, manual refcounting, custom allocators, `struct` composition). Critical hot paths include:
- **Command list recording**: `d3d12_command_list` — state tracking, root signature binding, descriptor heap management.
- **Descriptor heap emulation**: `d3d12_descriptor_heap`, descriptor copying/updating — the costliest CPU-side operation in many D3D12 titles.
- **Root signature and pipeline layout management**: Translation of D3D12 root parameters to Vulkan push constants and descriptor sets.
- **Barrier translation**: `d3d12_command_list_barrier` — resource state tracking and Vulkan barrier emission.
- **SPIR-V compilation**: `spirv_compiler` / `vkd3d_shader_compile` — DXIL→SPIR-V translation.
- **Command allocator and pool management**: Per-frame recycling of command buffers and descriptor pools.
- **Sparse binding and tiled resources**: If the file touches these, note that Vega's sparse residency support has specific page-size constraints.

# Contextual Hot-Path Patterns (Use Only If Relevant to This File)

Apply these patterns **only** when they match what the file actually does:

- **Per-draw CPU overhead**: In D3D11 translation (DXVK), each `Draw`/`DrawIndexed` call triggers dirty-flag evaluation, descriptor updates, pipeline binding, and dynamic state setting. Reducing branches and improving data locality in this path directly reduces CPU-bound frame times. The CS thread serialization/deserialization cost of each command is also critical — smaller `DxvkCsCmd` structs mean more commands per chunk, fewer chunk allocations.
- **Descriptor set management**: Frequent `vkUpdateDescriptorSets` calls with small write counts are costly. Batching writes, using `VK_KHR_push_descriptor` where RADV supports it efficiently, or reducing the number of unique descriptor set layouts can help. On RADV/ACO, descriptor loads from buffer addresses may scalarize better than descriptor set accesses.
- **Hash maps and caching**: DXVK uses `DxvkHashMap` (robin-hood) and `std::unordered_map` in various caches. Pipeline lookup, shader key hashing, and resource view caching are per-frame operations. Reduce hash collisions, avoid per-lookup allocations, and prefer open addressing with good load factors.
- **Barrier accumulation**: `DxvkBarrierSet` accumulates barriers and flushes them in batch. Over-synchronization (redundant barriers) wastes GPU time on Vega. Under-synchronization causes corruption. Barrier deduplication and layout-transition tracking improvements can reduce GPU stalls.
- **SPIR-V emission**: In `DxbcCompiler` (DXVK) or `spirv_compiler` (vkd3d-proton), the sequence of SPIR-V instructions emitted affects ACO's ability to scalarize, reduce VGPR pressure, and schedule instructions. Emitting uniform loads as `OpAccessChain` from UBO rather than SSBO can enable ACO's scalar load path on GFX9.
- **Memory allocation churn**: `std::vector` resize/realloc in per-draw paths, temporary `std::string` construction for debug names, and per-frame `new`/`delete` for transient objects all cause allocator contention and cache pollution.
- **Lock contention**: `std::mutex` or `pthread_mutex` in pipeline cache lookups, descriptor pool allocation, or shader module creation can serialize the CS thread against the app thread. Lock-free paths, reader-writer locks, or lock amortization help.
- **Branch-heavy polymorphism**: Virtual method calls in tight loops (e.g., resource type dispatch) cause indirect-branch mispredictions on Raptor Lake (~15–20 cycle penalty per mispredict). `switch` on enum or template dispatch may be faster when the type set is small and stable.
