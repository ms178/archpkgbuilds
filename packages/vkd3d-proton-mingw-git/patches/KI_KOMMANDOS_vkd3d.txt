PROJECT ARCHITECTURE -- READ BEFORE ANALYZING
========================================================================

vkd3d-proton translates the Direct3D 12 API into Vulkan. It is built as a Windows DLL (d3d12.dll) via MinGW cross-compilation, then loaded by Wine/Proton as a drop-in replacement for the Windows D3D12 runtime. The codebase is primarily C (GNU C17 with extensions) with select C++ components for shader compilation.

COM Implementation in C:
  vkd3d-proton implements D3D12 COM interfaces in C using manually constructed vtable structs (e.g., ID3D12DeviceVtbl, ID3D12GraphicsCommandListVtbl). The impl_from_ID3D12Device() pattern recovers the concrete struct from an interface pointer via CONTAINING_RECORD / container_of. Reference counting uses InterlockedIncrement/InterlockedDecrement (via compiler intrinsics on MinGW). QueryInterface, AddRef, and Release must remain strictly correct.

Key Subsystems and Hot Paths:

  Command List Recording (command.c and related):
    d3d12_command_list is the central recording structure. Every D3D12 draw, dispatch, copy, barrier, and state-setting call flows through it. The struct is large (several KB) and contains graphics and compute pipeline state, dirty flags, bound root signature state, descriptor heap bindings, render target state, and Vulkan command buffer handles. This is the single hottest CPU path in most D3D12 titles. Per-draw cost includes dirty-flag evaluation, root parameter rebinding, descriptor set updates, dynamic state emission, and VkCmd* recording.

  Descriptor Heap Emulation (resource.c, command.c):
    D3D12 uses shader-visible descriptor heaps (CBV/SRV/UAV and sampler). vkd3d-proton must translate these into Vulkan descriptors. Descriptor copying (CopyDescriptorsSimple, CopyDescriptors) is extremely hot in many titles. The translation involves mapping D3D12 descriptor indices to Vulkan descriptor set entries or buffer device addresses. On RADV, VK_EXT_descriptor_buffer or bindless descriptor indexing may be used depending on the code path.

  Root Signature and Pipeline Layout (state.c):
    D3D12 root signatures define the binding layout: root constants (mapped to Vulkan push constants), root CBV/SRV/UAV (mapped to push descriptors or dynamic buffer addresses), and descriptor tables (mapped to Vulkan descriptor sets). Root signature compilation and the per-draw root parameter binding path are performance-critical.

  Barrier Translation (command.c):
    D3D12 resource barriers (transition, UAV, aliasing) must be translated to Vulkan pipeline barriers (vkCmdPipelineBarrier2). Resource state tracking, barrier batching, and transition deduplication occur here. Over-synchronization wastes GPU cycles on Vega; under-synchronization causes corruption.

  Pipeline State Object (PSO) Management (state.c):
    d3d12_pipeline_state wraps Vulkan graphics and compute pipelines. PSO creation involves shader compilation (DXIL to SPIR-V via vkd3d-shader), pipeline layout construction, and VkPipeline creation. Pipeline caching and lookup cost affect both compilation stutter and steady-state overhead.

  Shader Compilation (libs/vkd3d-shader/):
    DXIL (and sometimes DXBC) bytecode is compiled to SPIR-V. The generated SPIR-V is then compiled by RADV's ACO backend into GFX9 ISA. The quality of generated SPIR-V directly affects ACO's ability to scalarize loads, reduce VGPR pressure, and schedule instructions efficiently on Vega 64.

  Command Queue and Submission (command.c):
    d3d12_command_queue manages a submission thread that serializes command buffer submission, fence signaling (via timeline semaphores), and present operations. The submission thread must stay ahead of the GPU; if it stalls, GPU bubbles appear.

  Memory and Resource Management (resource.c):
    D3D12 heaps and resources map to VkDeviceMemory and VkBuffer/VkImage. Placed, committed, and reserved (tiled/sparse) resources each have different allocation paths. Vega's sparse residency has 64 KB page granularity.

  Swapchain (swapchain.c):
    DXGI swapchain translation to VkSwapchainKHR, including present mode selection and frame latency management.

Threading Model:
  - Application threads call D3D12 API entry points and record command lists. Multiple command lists may be recorded concurrently on different threads.
  - A dedicated submission thread (per command queue) handles ExecuteCommandLists, fence signaling, and present.
  - Pipeline compilation may use worker threads (thread pool) for async PSO creation.
  - Some shared caches (pipeline cache, descriptor pool free-lists) require synchronization, typically via SRW locks (Slim Reader/Writer locks on Windows/MinGW) or mutexes via vkd3d_mutex.

HOT-PATH PATTERNS -- APPLY ONLY IF RELEVANT TO THIS FILE
========================================================================

These patterns describe known performance-sensitive areas in vkd3d-proton. Apply only when the provided file touches these areas.

Per-Draw/Dispatch CPU Overhead:
  Each D3D12 Draw/Dispatch call on a command list triggers dirty-flag evaluation in d3d12_command_list, checking which state needs re-emission. This includes root parameter rebinding (push constants, descriptor sets), pipeline binding, dynamic state (viewports, scissors, blend constants, stencil ref), vertex buffer binding, index buffer binding, and render target state. The dirty flags are typically a bitmask checked with bitwise AND. Reducing the number of branches in dirty-flag evaluation and improving data locality of the state being checked directly reduces CPU-bound frame times. At 30,000-80,000 draws per frame in heavy D3D12 titles, even 10-20 ns savings per draw translates to 0.3-1.6 ms per frame.

Descriptor Heap Operations:
  CopyDescriptorsSimple and CopyDescriptors are the D3D12 descriptor copy entry points. Many engines copy thousands of descriptors per frame (sometimes per draw). The inner loop copies descriptor data from source to destination heap entries. Each descriptor entry is typically 32-64 bytes (containing VkDescriptorBufferInfo, VkDescriptorImageInfo, or VkBufferView plus metadata). Optimizing this copy loop for cache line alignment and SIMD copying can yield large wins. On RADV with descriptor buffers, the descriptors may be written directly to mapped GPU memory, making write-combining behavior relevant.

Root Signature Binding:
  When the root signature changes (less frequent) or root parameters are updated (per-draw in many engines), vkd3d-proton must translate root CBV/SRV/UAV parameters into Vulkan push descriptors or buffer device addresses, root constants into push constant updates, and descriptor table bindings into Vulkan descriptor set binds. The cost depends on the root signature complexity. Engines like Unreal Engine 5 use relatively simple root signatures; proprietary engines may use complex ones with many root parameters.

Barrier Translation:
  D3D12 ResourceBarrier calls are batched into a single Vulkan vkCmdPipelineBarrier2 where possible. The translation must map D3D12 resource states (D3D12_RESOURCE_STATE_*) to Vulkan stage/access/layout combinations. Barrier deduplication (avoiding redundant transitions when a resource is already in the correct state) saves GPU time. On Vega 64, unnecessary barriers cause command processor stalls and wave drain, which are especially costly because Vega has 64 CUs that must all drain.

Pipeline Compilation Stutter:
  First-time PSO creation involves DXIL-to-SPIR-V compilation (via vkd3d-shader) followed by VkPipeline creation (which triggers ACO compilation in RADV). This is the primary source of shader compilation stutter. Pipeline caching, background compilation, and reducing the cost of pipeline key hashing/comparison all help. The shader cache key must be fast to compute and compare because it is evaluated on every PSO lookup.

Hash Map and Cache Lookups:
  vkd3d-proton uses hash maps for pipeline lookups, shader caches, format conversion tables, and descriptor layout caches. Hash function quality (collision rate, distribution), hash map load factor, and key comparison cost all affect per-frame overhead. Robin-hood hashing or open addressing with linear probing tends to be cache-friendly on modern CPUs.

Memory Allocation Patterns:
  Per-draw or per-frame allocations (vkd3d_malloc in hot paths, dynamic array growth, temporary buffer creation) cause allocator contention and cache pollution. Object pooling, arena allocation, and pre-sized arrays reduce this overhead. On MinGW, malloc may go through MSVCRT or UCRT, which have different performance characteristics than glibc malloc.

Lock Contention:
  SRW locks and mutexes protecting shared caches (pipeline cache, descriptor pool free list, memory allocator) can serialize the application thread against the submission thread or pipeline compilation threads. Reducing lock scope, using try-lock patterns, read-write lock differentiation (shared for lookup, exclusive for insert), or lock-free data structures can reduce contention. False sharing on lock-adjacent data is common.

SPIR-V Quality for ACO/GFX9:
  Generated SPIR-V that uses OpAccessChain into UBO (uniform buffer objects) instead of SSBO for constant data enables ACO to emit scalar memory loads (SMEM) instead of vector memory loads (VMEM) on GFX9. SMEM loads have ~80-cycle latency vs ~300-400 cycles for VMEM uncached. Uniform control flow, convergent operations, and minimal VGPR usage enable higher wave occupancy on Vega (occupancy is limited by max(VGPRs, SGPRs) per SIMD unit).
